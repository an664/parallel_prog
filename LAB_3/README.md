## Задание

## Теоретические сведения

MPI используется для параллельного программирования в модели распределенной
памяти. В отличие от OpenMP, исполнители являются отдельными процессами, и обмен
данными между ними выполняется явно через сообщения. Такой подход подходит не
только для одного компьютера, но и для кластеров.

В этой работе матрица `A` делится по строкам. Каждый процесс получает свой набор
строк `A`, полную матрицу `B` и вычисляет соответствующие строки результата
`C`. Использованные коллективные операции:

- `MPI_Bcast` - рассылка размера задачи и матрицы `B` всем процессам;
- `MPI_Scatterv` - распределение строк `A`, в том числе при неравном делении;
- `MPI_Gatherv` - сбор строк результата `C` на процессе с рангом 0.

Ускорение MPI ограничивается не только вычислениями, но и временем рассылки
данных, запуском процессов и сбором результата. Поэтому для маленьких матриц
накладные расходы могут быть сопоставимы с полезной работой.

## Реализация

Основная программа находится в [main.cpp](./main.cpp). Процесс с рангом 0
читает исходные матрицы. Матрица `B` рассылается всем процессам через
`MPI_Bcast`, строки матрицы `A` распределяются через `MPI_Scatterv`, а строки
результата собираются через `MPI_Gatherv`.

Такой способ деления задачи удобен для матриц, размер которых не обязан делиться
на количество процессов без остатка.

## Запуск

```bash
make
mpirun --map-by slot --oversubscribe -np 4 ./matrix_mpi sample_A.txt sample_B.txt result.txt
python3 verify.py sample_A.txt sample_B.txt result.txt
```

Полный эксперимент:

```bash
python3 benchmark.py
python3 plot_results.py
```

Каждая пара `размер матрицы / число MPI-процессов` запускается 3 раза. В
таблице и на графиках используется медиана времени. Полные агрегированные
данные сохранены в [results.csv](./results.csv).

## Верификация

## Выводы

MPI-версия корректно распределяет строки матрицы между процессами и показывает
ускорение на больших размерах. Для `N=2000` время уменьшилось с `7.404391` с
на одном процессе до `1.138494` с на 8 процессах, ускорение составило примерно
`6.50x`.

На этой локальной машине лучшая OpenMP-версия в большинстве случаев немного
быстрее лучшей MPI-версии. Это ожидаемо: OpenMP работает в одном процессе с
общей памятью, а MPI дополнительно тратит время на запуск процессов и
коллективные операции. При этом MPI важен как более общий подход: та же модель
может быть перенесена на несколько узлов кластера, где OpenMP уже недостаточен.

Повторные запуски сглаживают отдельные выбросы `mpirun` и дают более надежную
картину масштабирования. Для небольших матриц результаты особенно чувствительны
к накладным расходам, поэтому основное сравнение лучше делать на `N=1200` и
выше.
